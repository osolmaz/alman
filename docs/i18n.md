# Internationalization

How the site's language versions are identified and served.

## Locales

| Locale | Language | URL prefix | BCP 47 tag | Spec i18n key |
|--------|----------|------------|------------|---------------|
| `en` | English (canonical) | `/` | `en` | — (top-level fields) |
| `de` | German | `/de/` | `de` | `de` |
| `al` | Alman | `/al/` | `de-AL` | `al` |

## Alman's language code

Alman's BCP 47 language tag is **`de-AL`**. The project petitions for this
tag to be recognized officially and uses it in markup (`lang`, `hreflang`)
as if it already were. Do not use private-use tags such as `de-x-alman`.

In URLs and internally (locale keys, spec `i18n` blocks, content file
names), the bare **`al`** is used instead, because it is shorter and
simpler.

## Where translations live

- **Spec**: `i18n` objects with `de` and `al` keys next to the English
  prose fields in the JSON files under `spec/`. `uv run python -m
  alman.build` generates `_includes/spec.md`, `spec.de.md`, and
  `spec.al.md` from them.
- **About page**: `src/content/about/{en,de,al}.md`. The `de` and `al`
  versions must stay structurally parallel — the Alman page renders a
  word-level diff against the German one.
- **UI strings**: `src/i18n/ui.ts`, which also defines the locale
  metadata used by `astro.config.mjs` and the navbar language switcher.

Translation style is governed by the `translation-style` skill in
`.agents/skills/translation-style/SKILL.md`: no language purism, faithful
to the English source, and Alman text must comply with the Alman grammar.
